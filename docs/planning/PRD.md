# 이벤트 관리 플랫폼 MVP PRD

## 핵심 정보

**목적**: 생일파티, 워크샵, 친구 모임 등 일회성 이벤트에서 발생하는 공지·참여자 관리·카풀·정산 부담을 하나의 서비스로 통합 해결
**사용자**: 소규모 일회성 이벤트를 주최하거나 참여하는 개인 사용자 (모바일 중심, 관리자는 PC)

---

## 사용자 여정

### 주최자 여정

```
1. 로그인 페이지 (이메일 또는 Google OAuth)
   ↓ 로그인 성공 → role 확인

2. 주최자 홈 (내 이벤트 요약 카드)
   ↓ "이벤트 만들기" 버튼 클릭

3. 이벤트 생성 페이지 (제목, 카테고리, 일시, 장소, 인원, 공개 여부 입력)
   ↓ 생성 완료 → 자동 리디렉션

4. 이벤트 상세 페이지 (주최자 뷰) — 초대 링크 복사, 탭 탐색
   ↓ 탭 선택에 따라 분기

   [참여자] → 참여자 관리 페이지 → 승인/거절/출석 체크
   [공지/댓글] → 공지 및 댓글 관리 페이지 → 공지 작성/댓글 관리
   [카풀] → 카풀 관리 페이지 → 카풀 등록/신청 승인
   [정산] → 정산 관리 페이지 → 항목 입력 → 1/N 결과 확인
   ↓

5. 이벤트 수정/삭제 (필요 시)
```

### 참여자 여정

```
1. 로그인 페이지
   ↓ 로그인 성공 → role 확인

2. 이벤트 탐색 페이지 (카드 목록, 카테고리 필터)
   또는 초대 링크 직접 접근 → 이벤트 상세 페이지 (참여자 뷰)
   ↓ "참여 신청" 버튼 클릭

3. 참여 신청 처리
   ↓ 신청 완료 → 승인 대기 상태

   [승인 전] → 내가 참여한 이벤트 목록 (pending 상태 표시)
   [승인 후] → 이벤트 상세 접근 가능 → 카풀/정산 탭 노출
   ↓

4. 카풀 탑승 신청 페이지 (승인된 카풀에 탑승 요청)

5. 정산 현황 페이지 (내가 낼 금액 확인)
```

### 관리자 여정

```
1. 로그인 페이지
   ↓ 로그인 성공 → role = 'admin' 확인 → /admin 리디렉션

2. 관리자 대시보드 (KPI 카드 4개 + 이벤트 현황 차트 + 최근 가입자 테이블)
   ↓ 사이드바 메뉴 선택

   [이벤트 관리] → 전체 이벤트 목록 및 관리 페이지
   [사용자 관리] → 사용자 목록 및 역할 변경 페이지
```

## 메뉴 구조

```
인증 메뉴 (비로그인 사용자)
├── 로그인 — F010
└── 회원가입 — F010

주최자 메뉴 (로그인 후, role = 'user', 이벤트 생성 시 host 역할 부여)
하단 탭 내비게이션
├── 홈 — F001 (내 이벤트 요약)
├── 내 이벤트 — F001 (내가 만든 이벤트 목록)
├── 만들기 — F001 (이벤트 생성 폼)
└── 프로필 — F012 (프로필 조회/수정)

이벤트 상세 탭 (주최자 전용)
├── 상세 정보 — F001, F007 (이벤트 정보, 초대 링크)
├── 수정 — F001 (이벤트 수정 폼)
├── 참여자 — F002 (승인/거절/출석 체크)
├── 공지/댓글 — F003 (공지 작성, 댓글 관리)
├── 카풀 — F004 (카풀 등록, 신청 승인)
└── 정산 — F005 (항목 입력, 결과 확인)

참여자 메뉴 (로그인 후)
하단 탭 내비게이션
├── 탐색 — F006 (이벤트 카드 목록, 카테고리 필터)
├── 참여중 — F002 (내가 참여한 이벤트 목록)
├── 카풀 — F004 (탑승 신청한 카풀 목록)
└── 프로필 — F012 (프로필 조회/수정)

이벤트 상세 탭 (참여자 전용)
├── 상세 정보 — F002, F003 (이벤트 정보, 참여 신청, 댓글)
├── 카풀 — F004 (카풀 탑승 신청)
└── 정산 — F005 (내 정산 현황)

관리자 메뉴 (role = 'admin')
상단 GNB + 좌측 사이드바
├── 대시보드 — F008 (KPI 카드, 이벤트 차트, 최근 가입자)
├── 이벤트 관리 — F009 (전체 이벤트 목록/관리)
└── 사용자 관리 — F009 (사용자 목록/역할 변경)

공통
└── 로그아웃 — F010
```

## 페이지별 상세 기능

### 루트 페이지

> **구현 기능:** `F011` | **접근:** 서비스 진입점

| 항목            | 내용                                                                                                                      |
| --------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **역할**        | 로그인 상태 및 role을 확인하여 적절한 페이지로 자동 리디렉션                                                              |
| **진입 경로**   | 서비스 루트 URL 직접 접근                                                                                                 |
| **사용자 행동** | 별도 행동 없음 — 자동 리디렉션 처리                                                                                       |
| **주요 기능**   | 비로그인 → 로그인 페이지 리디렉션, role = 'admin' → 관리자 대시보드 리디렉션, role = 'user' → 이벤트 탐색 페이지 리디렉션 |
| **다음 이동**   | 비로그인 → 로그인 페이지, admin → 관리자 대시보드 페이지, user → 이벤트 탐색 페이지                                       |

| **다음 이동**   | 성공 → role에 따라 이벤트 탐색 페이지 또는 관리자 대시보드 페이지, 실패 → 에러 메시지 표시                                 |

## 정산 알고리즘 명세

### calculateSettlement 로직

```
입력: event_id
출력: 최소 거래 쌍 목록 [{ from: 사용자명, to: 사용자명, amount: 금액 }]

1단계 — 총 지출 집계
  SELECT SUM(amount) FROM settlement_items WHERE event_id = :event_id

2단계 — 참여자 목록 조회 (주최자 포함)
  host + SELECT user_id FROM participations WHERE event_id = :event_id AND status = 'approved'
  참여자 수 N = 위 목록 길이

3단계 — 1인 균등 부담액 계산
  perPerson = Math.ceil(totalAmount / N)  (원 단위 올림)

4단계 — 각 참여자 순잔액 계산
  paid = 해당 참여자가 settlement_items에서 지출한 총액 (없으면 0)
  balance = paid - perPerson
  balance > 0: 받을 돈 (creditor)
  balance < 0: 낼 돈 (debtor)
  balance = 0: 정산 완료

5단계 — 그리디 알고리즘으로 최소 거래 쌍 결정
  creditors 배열 (balance > 0, 큰 순)
  debtors 배열 (balance < 0, 절댓값 큰 순)

  while creditors.length > 0 and debtors.length > 0:
    creditor = creditors[0]
    debtor   = debtors[0]
    amount   = min(creditor.balance, abs(debtor.balance))
    거래 기록: debtor.name → creditor.name, amount
    creditor.balance -= amount
    debtor.balance   += amount
    balance = 0인 항목 제거

출력: 거래 쌍 목록
```

## 아키텍처 설계

### 역할 기반 접근 제어 흐름

```
요청 진입
  ↓
미들웨어 (proxy.ts — updateSession)
  ↓ 세션 없음
  → /auth/login 리디렉션
  ↓ 세션 있음
  → /admin/* 접근 시 role = 'admin' 체크 → 아닌 경우 홈 리디렉션
  → /(host)/* 접근 시 인증 확인
  → /(participant)/* 접근 시 인증 확인
```

### 주최자 역할 부여 방식

- 전체 사용자는 DB role = 'user'로 동일
- 이벤트를 생성한 사용자가 해당 이벤트의 host_id를 보유
- 주최자 뷰(/host/\*)는 role = 'user'인 인증 사용자 모두 접근 가능
- 이벤트 상세/편집/관리 페이지는 host_id = auth.uid() 조건으로 접근 제어
- 관리자(role = 'admin')만 /admin/\* 접근 가능

## 구현 Phase

### Phase 1 — 기반 구축 + 이벤트 CRUD

> ⚠️ 특이사항
>
> 1. **[미들웨어 admin 접근 제어]** `proxy.ts`(미들웨어)에 `/admin/*` 경로 접근 시 DB에서 role을 확인한 후 리디렉션하는 로직을 반드시 추가해야 함. RLS 정책만으로는 admin 페이지 자체 렌더링을 막을 수 없음.
> 2. **[프로필 스키마 마이그레이션]** Migration 001 실행 전, 현재 profiles 테이블과 목표 스키마 간 차이를 명확히 파악하고 마이그레이션 SQL을 신중히 작성할 것. role 컬럼 부재로 인증 시스템 전체가 동작하지 않을 수 있음.

- 패키지 설치 (react-hook-form, zod, date-fns, recharts)
- TailwindCSS v4 업그레이드
  - 현재 설치된 TailwindCSS v3 → v4로 업그레이드 필요
  - 설치 명령어: `npm install tailwindcss@next @tailwindcss/postcss@next`
  - v4에서는 `tailwind.config.ts` 대신 CSS 파일에서 직접 설정하는 방식으로 변경됨
  - `globals.css` 최상단에 `@import "tailwindcss"` 를 추가하는 방식으로 변경됨 (`@tailwind base/components/utilities` 지시어 제거)
- shadcn 컴포넌트 추가 (textarea, select, separator, tabs, avatar, toast, skeleton, dialog, sheet)

### Phase 2 — 참여자 관리 + 공지/댓글

> ⚠️ 특이사항
>
> 1. **[UNIQUE 제약 및 인덱스]** Migration 003에서 participations 테이블에 `(event_id, user_id)` UNIQUE 제약을 반드시 추가해야 함. 미추가 시 동일 사용자가 동일 이벤트에 중복 신청 가능. 추가 권장 인덱스: `participations(event_id)`, `participations(user_id, status)`.
> 2. **[참여 승인 동시성 제어]** 여러 참여 신청을 동시에 승인할 때 `max_participants`를 초과하지 않도록 Supabase RPC(PostgreSQL 함수)로 원자적 처리가 필요함. 일반 UPDATE로는 Race Condition이 발생할 수 있음.
> 3. **[ON DELETE CASCADE]** Migration 003에서 participations 테이블의 event_id 외래키에 `ON DELETE CASCADE`를 설정하여 이벤트 삭제 시 연관 참여 데이터가 자동 삭제되도록 해야 함.

- Migration 003, 004 실행 + 타입 재생성
- 참여 신청 → 승인/거절 → 출석 체크 전체 흐름
- 공지/댓글 피드 통합 (공지 배지 강조)
- 초대 링크 공유 기능

**검증 기준**: 참여 신청 전체 흐름 + 공지/댓글 작성 동작

### Phase 3 — 카풀 기능

> ⚠️ 특이사항
>
> 1. **[카풀 UNIQUE 제약 및 인덱스]** Migration 005에서 carpool_requests 테이블에 `(carpool_id, passenger_id)` UNIQUE 제약을 반드시 추가해야 함. 미추가 시 동일 탑승 신청이 중복으로 등록될 수 있음. 추가 권장 인덱스: `carpools(event_id)`, `carpool_requests(carpool_id)`, `carpool_requests(passenger_id)`.
> 2. **[카풀 좌석 동시성 제어]** 탑승 승인 시 `total_seats`를 초과하지 않도록 Supabase RPC(PostgreSQL 함수)로 원자적 처리가 필요함. 현재 승인된 탑승자 수를 SELECT 후 UPDATE하는 방식은 Race Condition에 취약함.
> 3. **[ON DELETE CASCADE]** carpools의 event_id, carpool_requests의 carpool_id 외래키에 `ON DELETE CASCADE` 설정 필요.

- Migration 005 실행 + 타입 재생성

**검증 기준**: 카풀 등록 → 신청 → 승인/거절 → 좌석 현황 확인

### Phase 4 — 정산 + 관리자 대시보드

> ⚠️ 특이사항
>
> 1. **[정산 올림 처리 보정]** `calculateSettlement` 함수에서 `Math.ceil(totalAmount / N)` 사용 시 총 부담액이 실제 지출보다 커지는 문제가 있음. 예) 10,000원 / 3명 = 3,334원 × 3 = 10,002원. `Math.floor` 사용 후 나머지 금액을 첫 번째 debtor에게 할당하거나, 마지막 참여자에게 나머지를 배분하는 방식으로 보정할 것.
> 2. **[ON DELETE CASCADE]** settlement_items의 event_id 외래키에 `ON DELETE CASCADE` 설정 필요.
> 3. **[인덱스]** Migration 006에서 `settlement_items(event_id)` 인덱스 추가 권장.
