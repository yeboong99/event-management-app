# 카풀 기능 E2E 테스트 계획 (Playwright MCP)

## Context

카풀 기능 구현(Task 65)이 완료된 상태로, 다음 기능들이 정상 동작하는지 E2E 테스트를 수행합니다:

- 카풀 등록/삭제
- 탑승 신청/승인/거절/취소
- 이벤트 내 카풀 1개 제한 (CARPOOL_CONFLICT 다이얼로그)
- rejected 상태에서 다른 카풀 신청 차단
- "내 카풀" 탭에서 모든 상태(pending/approved/rejected) 취소 버튼 표시
- "카풀 목록" 탭에서는 pending 상태만 취소 버튼 표시

## 테스트 계정

| 계정                  | 역할          |
| --------------------- | ------------- |
| testuser1@example.com | 이벤트 호스트 |
| testuser2@example.com | 카풀 드라이버 |
| testuser4@example.com | 탑승 신청자   |

비밀번호: `Test1234!` (공통)

## 앱 URL

개발 서버: `http://localhost:3000`

## 사전 준비 (Setup)

앱 URL: `http://localhost:3000`

### Setup-1: testuser1으로 새 이벤트 생성

1. testuser1 로그인
2. 이벤트 생성 (`/events/new`)
   - 제목: "카풀 테스트 이벤트"
   - 날짜: 미래 날짜
   - 최대 참여자: 10명
   - 참여 신청 필요 설정
3. 생성된 이벤트 URL 기록

### Setup-2: testuser2, testuser4 참여 신청 및 승인

1. testuser2 로그인 → 이벤트 참여 신청
2. testuser4 로그인 → 이벤트 참여 신청
3. testuser1 로그인 → 참여자 탭에서 testuser2, testuser4 승인

---

## 시나리오 목록

### [S1] 카풀 등록 기본 플로우

**목적:** 승인된 참여자가 카풀을 등록할 수 있는지 확인

**실행 계정:** testuser2

**단계:**

1. 이벤트 상세 페이지 → "카풀" 탭 이동
2. "카풀 만들기" 버튼 클릭 → 등록 폼 인라인 표시 확인
3. 출발지, 출발 시각, 좌석 수 (본인 포함) 입력하여 카풀 등록
4. "카풀 목록" 탭에 등록된 카풀 카드 표시 확인
5. "내 카풀" 탭에도 해당 카풀 표시 확인
6. "카풀 목록" 탭의 카운트 뱃지가 1로 표시되는지 확인

**기대 결과:** 카풀 카드가 두 탭 모두에 표시되고, 카운트 뱃지가 올바르게 갱신됨

---

### [S2] 탑승 신청 및 승인 플로우

**목적:** 탑승 신청 → 드라이버 승인 → 상태 변경 정상 동작 확인

**실행 계정:** testuser4 (신청), testuser2 (승인)

**단계:**

1. testuser4 로그인 → 이벤트 카풀 탭 이동
2. testuser2의 카풀 카드에서 탑승 신청 폼 확인
3. 메시지 입력 후 "탑승 신청" 버튼 클릭
4. 신청 상태 "대기 중" Badge 표시 확인
5. testuser2 로그인 → 이벤트 카풀 탭 이동
6. 카풀 카드의 "탑승 신청 관리" 섹션에서 testuser4 신청 확인
7. "승인" 버튼 클릭
8. testuser4 로그인 → "내 카풀" 탭에서 "승인됨" Badge 확인

**기대 결과:** 상태가 "대기 중" → "승인됨"으로 변경되고, 승인 후 승인된 탑승자 수(approved_count) 반영

---

### [S3] 이벤트 내 카풀 1개 제한 — pending 상태에서 다른 카풀 신청 차단

**목적:** 이미 pending 신청이 있을 때 다른 카풀 신청 시 CARPOOL_CONFLICT 다이얼로그 표시

**사전 조건:** testuser4가 testuser2 카풀에 pending 상태 신청 존재

**실행 계정:** testuser4

**단계:**

1. testuser4 로그인 → 카풀 탭
2. 두 번째 카풀이 존재하는 상태에서 (testuser1이 추가 카풀 등록) 해당 카풀의 탑승 신청 폼에서 신청 시도
3. CARPOOL_CONFLICT 안내 다이얼로그 표시 확인
4. 다이얼로그에 "내 카풀 탭에서 기존 신청을 취소" 안내 메시지 포함 여부 확인
5. "확인" 버튼으로 다이얼로그 닫기

**기대 결과:** 신청이 차단되고 CARPOOL_CONFLICT 다이얼로그가 표시됨

---

### [S4] 탑승 신청 거절 + rejected 상태 차단 + 취소 후 재신청

**목적:** rejected 상태 사용자가 취소 전까지 다른 카풀 신청 불가, 취소 후 재신청 가능

**실행 계정:** testuser2 (거절), testuser4 (재신청)

**단계:**

1. testuser2가 testuser4의 신청 거절
2. testuser4 로그인 → "카풀 목록" 탭에서 "거절됨" Badge 확인
3. testuser4가 다른 카풀에 신청 시도 → CARPOOL_CONFLICT 다이얼로그 확인
4. testuser4가 "내 카풀" 탭으로 이동
5. **"거절됨" Badge 우측에 "신청 취소" 버튼 표시 확인** (핵심 검증)
6. "신청 취소" 버튼 클릭 → 신청이 "내 카풀" 탭에서 사라짐 확인
7. testuser2의 탑승 신청 관리 목록에서도 사라짐 확인
8. testuser4가 다른 카풀에 신청 시도 → 이번에는 정상 신청 성공 확인

**기대 결과:** rejected 상태도 차단되고, 취소 후에만 재신청 가능

---

### [S5] 내 카풀 탭 vs 카풀 목록 탭 취소 버튼 가시성 비교

**목적:** `showCancel` prop 분기가 올바르게 동작하는지 확인

**실행 계정:** testuser4

**단계:**

**[S5-A] approved 상태 확인:**

1. testuser4가 승인된 상태에서 "카풀 목록" 탭 이동
2. 해당 카풀 카드에서 "신청 취소" 버튼 **미표시** 확인
3. "내 카풀" 탭 이동
4. 해당 카풀 카드에서 "신청 취소" 버튼 **표시** 확인

**[S5-B] rejected 상태 확인:**

1. testuser4가 거절된 상태에서 "카풀 목록" 탭 이동
2. 해당 카풀 카드에서 "신청 취소" 버튼 **미표시** 확인
3. "내 카풀" 탭 이동
4. 해당 카풀 카드에서 "신청 취소" 버튼 **표시** 확인

**기대 결과:** approved/rejected 상태에서 "내 카풀" 탭에서만 취소 버튼 표시

---

### [S6] approved 상태 신청 취소

**목적:** 승인된 탑승자가 신청을 취소할 수 있고, 취소 후 재신청 가능 확인

**실행 계정:** testuser4

**단계:**

1. testuser4가 승인된 상태에서 "내 카풀" 탭 이동
2. "신청 취소" 버튼 클릭
3. "내 카풀" 탭에서 카드 사라짐 확인
4. testuser2의 탑승 신청 관리 목록에서도 사라짐 확인
5. testuser4가 다시 탑승 신청 폼 사용 가능 확인 (CARPOOL_CONFLICT 없이)

**기대 결과:** 승인된 상태에서도 취소 가능하고, 취소 후 재신청 폼이 정상 표시

---

### [S7] 카풀 생성 제한 — 이미 신청이 있을 때 카풀 생성 차단

**목적:** pending/approved/rejected 신청이 있는 상태에서 새 카풀 생성 시도 시 차단

**실행 계정:** testuser4

**단계:**

1. testuser4가 pending 신청 있는 상태에서 "카풀 만들기" 버튼 클릭
2. 카풀 등록 폼 제출 → CARPOOL_CONFLICT 다이얼로그 확인
3. 다이얼로그에 "내 카풀 탭에서 기존 신청을 취소 후 카풀을 등록" 안내 포함 여부 확인

**기대 결과:** 기존 신청이 있을 때 카풀 생성도 차단됨

---

### [S8] 카풀 삭제

**목적:** 드라이버가 자신의 카풀을 삭제할 수 있는지 확인

**실행 계정:** testuser2

**단계:**

1. testuser2 로그인 → 카풀 탭
2. 카풀 카드 우측의 삭제 버튼 클릭
3. 확인 다이얼로그 → 삭제 확인
4. 카풀이 "카풀 목록" 탭에서 사라짐 확인
5. 카운트 뱃지가 감소하는지 확인

**기대 결과:** 카풀 삭제 후 목록에서 사라지고 카운트 갱신

---

## 실행 순서 (의존성 고려)

```
Setup-1 → Setup-2
  → S1 (카풀 등록)
    → S2 (탑승 신청/승인)
      → S5-A (approved 취소 버튼 가시성)
        → S6 (approved 취소)
    → [S2 후 거절] → S4 (rejected 차단 + 취소 + 재신청)
      → S5-B (rejected 취소 버튼 가시성)
  → S3 (pending 상태 차단) — S1 후, testuser1이 두 번째 카풀 등록 필요
  → S7 (카풀 생성 제한) — pending 신청 있는 상태
  → S8 (카풀 삭제) — 마지막에 정리
```

## 검증 체크리스트

- [ ] S2: pending → approved 상태 전환 정상
- [ ] S3: pending 상태 → 다른 카풀 신청 시 CARPOOL_CONFLICT 다이얼로그
- [ ] S4: rejected 상태 → 다른 카풀 신청 시 CARPOOL_CONFLICT 다이얼로그
- [ ] S4: 취소 후 재신청 성공 (CARPOOL_CONFLICT 없음)
- [ ] S5-A: approved → "내 카풀" 탭만 취소 버튼, "카풀 목록" 탭 취소 버튼 없음
- [ ] S5-B: rejected → "내 카풀" 탭만 취소 버튼, "카풀 목록" 탭 취소 버튼 없음
- [ ] S6: approved 취소 후 신청 폼 재노출
- [ ] S7: 신청 있을 때 카풀 생성 차단
- [ ] S8: 카풀 삭제 후 목록에서 사라짐

## 실행 방법

Playwright MCP를 활용하여 각 시나리오를 순서대로 자동화 실행합니다.
개발 서버(`npm run dev`)가 `http://localhost:3000`에서 실행 중이어야 합니다.
