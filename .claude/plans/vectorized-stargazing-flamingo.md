# FLOW-001: Playwright MCP 통합 테스트

## Context

FLOW-001 구현이 완료되었습니다:

- `events.invite_token` 컬럼 추가 + `get_event_by_invite_token` RPC
- `/events/[eventId]/join?token={token}` 초대 랜딩 페이지
- 비공개 이벤트 `inviteUrl`을 토큰 기반 URL로 변경

이제 Playwright MCP를 이용해 브라우저에서 직접 전체 흐름을 검증합니다.

---

## 사전 조건

- 개발 서버: `http://localhost:3000` (실행 중이어야 함)
- 테스트 계정 2개 필요:
  - **계정A (주최자)**: 비공개 이벤트 생성 및 초대 링크 복사
  - **계정B (참여자)**: 초대 링크로 방문하여 참여 신청

---

## 테스트 시나리오 및 실행 순서

### Phase 0: 개발 서버 확인

Playwright로 `http://localhost:3000`에 접속하여 앱이 응답하는지 확인. 실패 시 중단하고 서버 기동 요청.

---

### Phase 1: 테스트 계정 준비

1. **계정A 생성/로그인**
   - `/auth/sign-up` → 이메일/비밀번호로 계정 생성 (또는 이미 있으면 로그인)
   - 테스트 이메일 예시: `host-test@example.com`

2. **계정B 생성**
   - 시크릿 탭이 불가능하므로, 계정A 작업 후 로그아웃 → 계정B 생성
   - 테스트 이메일 예시: `guest-test@example.com`

---

### Phase 2: 비공개 이벤트 생성 (계정A)

1. `/auth/login` → 계정A 로그인
2. `/events/new` 이동
3. 폼 입력:
   - 제목: "Playwright 테스트 비공개 이벤트"
   - 카테고리: 적당히 선택
   - 일시: 미래 날짜
   - **공개 여부 체크박스 해제** (비공개 설정)
4. 이벤트 생성 제출 → `/events/{eventId}` 로 이동 확인

---

### Phase 3: 초대 링크 복사 및 검증 (계정A)

1. 이벤트 상세 페이지에서 **"초대 링크 복사"** 버튼 클릭
2. Toast "초대 링크가 복사되었습니다" 확인
3. 복사된 URL이 `/events/{id}/join?token={uuid}` 형식인지 검증
   - Playwright로 클립보드 값 읽기: `navigator.clipboard.readText()`
   - UUID 패턴(`?token=`이 포함된 URL) 확인

**예상 URL 패턴:**

```
http://localhost:3000/events/{eventId}/join?token={uuid-v4}
```

---

### Phase 4: 초대 링크 에러 케이스 검증

다음 URL들을 직접 방문하여 에러 UI 표시 확인:

| 케이스         | URL                                                            | 예상 결과                   |
| -------------- | -------------------------------------------------------------- | --------------------------- |
| 토큰 없음      | `/events/{id}/join`                                            | 자물쇠 아이콘 + 에러 메시지 |
| 토큰 무효      | `/events/{id}/join?token=00000000-0000-0000-0000-000000000000` | 자물쇠 아이콘 + 에러 메시지 |
| eventId 불일치 | `/events/fake-id/join?token={실제토큰}`                        | 자물쇠 아이콘 + 에러 메시지 |

---

### Phase 5: 미인증 사용자 초대 링크 방문

1. 로그아웃 상태에서 복사한 초대 URL 방문
   - 미들웨어가 `/auth/login?redirectTo=...`로 리다이렉트함
   - 리다이렉트 URL에 `redirectTo` 파라미터가 포함되어 있는지 확인

> **참고**: 미들웨어가 미인증 사용자를 로그인으로 보내므로, 로그인 후 join 페이지로 돌아오는 흐름을 테스트.

---

### Phase 6: 참여 신청 전체 흐름 (계정B)

1. 계정A 로그아웃
2. 계정B로 로그인 (`/auth/login`)
3. 복사한 초대 URL(`/events/{id}/join?token={token}`) 방문
4. **Join 페이지 UI 확인**:
   - 이벤트 제목, 주최자명, 날짜, 장소 표시 확인
   - 카테고리 배지 + "비공개" 배지 표시 확인
   - ParticipationForm (신청 메시지 textarea + "참여 신청" 버튼) 표시 확인
5. 신청 메시지 입력 후 **"참여 신청"** 버튼 클릭
6. **성공 후 상태 변화 확인**:
   - Toast "참여 신청이 완료되었습니다" 표시
   - UI가 "신청 대기중" 배지 + 취소 버튼으로 변경
7. `/events/{id}` 직접 방문 시 404 확인 (RLS에 의해 차단)

---

### Phase 7: 참여 승인 (계정A)

1. 계정B 로그아웃
2. 계정A로 로그인
3. `/events/{id}` 방문 → 참여자 탭에서 계정B의 pending 신청 확인
4. **"승인"** 버튼 클릭 → approved 상태로 변경 확인

---

### Phase 8: 승인 후 흐름 검증 (계정B)

1. 계정A 로그아웃
2. 계정B로 로그인
3. 초대 URL 재방문
4. **상태 확인**:
   - "참여 승인됨 ✓" 배지 표시
   - "이벤트 페이지로 이동" 버튼 표시
5. 버튼 클릭 → `/events/{id}` 접근 성공 확인 (RLS 통과)

---

## 핵심 검증 포인트 요약

| #   | 검증 항목                         | 성공 기준                   |
| --- | --------------------------------- | --------------------------- |
| 1   | inviteUrl 형식                    | `?token={uuid}` 포함 여부   |
| 2   | 토큰 없음/무효                    | 자물쇠 아이콘 에러 페이지   |
| 3   | Join 페이지 기본 정보             | 이벤트 정보 정상 렌더링     |
| 4   | 미인증 접근                       | 로그인 후 join 페이지 복귀  |
| 5   | 참여 신청                         | pending 배지로 즉시 전환    |
| 6   | pending 상태에서 이벤트 직접 접근 | 404 반환                    |
| 7   | 승인 후 join 재방문               | approved 배지 + 이벤트 링크 |
| 8   | 승인 후 이벤트 직접 접근          | 정상 접근 (RLS 통과)        |

---

## 실행 시 주의사항

- **계정 전환**: 브라우저 세션이 하나이므로 로그아웃/로그인으로 계정 전환
- **클립보드 읽기**: `mcp__playwright__browser_evaluate`로 `navigator.clipboard.readText()` 실행 (또는 DOM에서 URL 추출)
- **개발 서버 미실행 시**: 사용자에게 `npm run dev` 실행을 요청
